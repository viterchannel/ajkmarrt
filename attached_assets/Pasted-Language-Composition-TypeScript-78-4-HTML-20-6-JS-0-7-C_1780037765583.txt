2.1 API Client Architecture ⭐ EXEMPLARY
File: api.ts (1252 lines of production code)

Token Management:
TypeScript
✅ Dual token storage:
   - Access token: Preferences (secure) + in-memory cache
   - Refresh token: HttpOnly cookie + Preferences
   - Migration: localStorage → Preferences on first boot

✅ BroadcastChannel for cross-tab sync:
   - Other tabs notified when token refreshes
   - Prevents 401 cascade across browser tabs

✅ Token refresh with mutex-guard:
   - Prevents concurrent refresh race conditions
   - api.refreshToken() exposed for manual refresh
CSRF Protection:
TypeScript
✅ Multi-layer CSRF strategy:
   1. Read from Preferences (primary)
   2. Fallback to document.cookie
   3. Extracted from auth response body
   4. Attached to all state-mutating requests
Error Handling:
TypeScript
✅ Custom 403 handling:
   - Distinguishes auth failures from business-rule blocks
   - APPROVAL_PENDING vs. APPROVAL_REJECTED (separate flows)
   - Extracted rejection reason & verification requirements

✅ Circuit breaker pattern:
   - Per-endpoint retries (CB_DEFAULT_RETRIES = 3)
   - Exponential backoff for 5xx errors
   - Cool-down: 30 seconds

✅ 401 → Refresh flow:
   - Mutex-guarded to prevent thundering herd
   - Token refresh callbacks fired on success
   - Automatic retry on subsequent 401
Security Measures:
TypeScript
✅ Attestation token support:
   - Play Integrity (Android)
   - App Attest (iOS)
   - Injected into X-Attest-Token header

✅ Session attestation validation:
   - Server validates device tokens on sensitive endpoints
   - Zero latency fallback on web (getAttestToken() returns "")

✅ Timeout management:
   - Configurable via setApiTimeoutMs()
   - Platform config driven
   - Default: 30s, max: 300s

TypeScript

   - JWT pattern: eyJ[...]{8+}.[...]{8+}.[...]{8+}
   - Query tokens: token=, access_token=, api_key=
